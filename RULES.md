# Rules

These rules apply to everything in this repository: code, documentation, UI copy, commit messages, and issues. They apply to every contributor, human or AI assistant. If a tool or assistant cannot follow these rules, do not use it here.

## Writing and copy

- No emojis anywhere. Not in docs, UI copy, code comments, commit messages, or issues.
- No em dashes. Do not use the long dash character.
- No double hyphens used as a dash. Do not write two hyphens in a row in prose to stand in for a dash. Rewrite with a comma, a colon, parentheses, or a new sentence. This applies to prose only. Command line flags in code or code spans, such as a flag written with two leading hyphens, are fine because they are code.
- No filler adverbs used for emphasis, in particular genuinely, honestly, and actually.
- No "not X, it is Y" or "not just X but Y" constructions.
- No AI tell tale phrasing. Avoid the patterns that read as machine generated, including but not limited to: delve, tapestry, seamless, robust as filler, leverage as a verb for use, unleash, elevate, supercharge, game changer, in today's fast paced world, it is worth noting, it is important to note, that said used as a transition crutch, moreover and furthermore stacked together, in conclusion, embark, navigate the complexities, testament to, nestled, boasts, at the end of the day, dive in, and rhetorical questions used to open a section.
- Do not stack bold text through prose. Use plain sentences. Headings and code carry the structure.
- Be concise. Say the thing once, plainly.
- Do not cite project counts or a number of published projects as a credential in marketing, pitch, or site copy.
- Do not describe the aesthetic as neon, synthwave, or dark. The look is bright, bold, playful arcade candy with deep ink outlines and warm paper.

## Code

- Client side first. The app runs in the browser with no runtime build step. Heavy or optional paths feature detect and fall back to a working path.
- Real methods only. Do not ship a capability that has no working implementation. If a technique cannot run in the target environment, remove the option rather than fake it.
- Verify against the real source. Confirm a library's actual interface before using it. Do not code against a remembered API.
- Small libraries, thin app. Put logic in framework agnostic packages. Keep screens limited to the DOM and to wiring packages together. Logic packages do not touch the DOM.
- Engineered well, not over engineered. Group by cohesive responsibility. Do not split into micro packages that add ceremony without payoff. Do not add a framework where a function will do.
- Stable interfaces. Each package has a documented public interface and a README. Changes to a public interface are deliberate and noted.
- Validate before done. Run the test suites (`npm test` and `npm run test:web`) and a production build (`npm run build`) before treating a change as finished.
- Accessibility is required. Every action reachable by keyboard. Respect reduced motion. Maintain contrast and visible focus.
- Comments explain why, not what. Keep them short and honest.

## Git

- Write commit subjects in the imperative mood, for example "add the editor package" not "added" or "adds".
- Keep the subject short and specific. Add a body only when it helps.
- No emojis in commit messages.
- No AI attribution of any kind in commits. Do not add a generated by line. Do not add a co author trailer for an AI. Never add Claude or any assistant as a co author. Do not mention the tool that produced the change.
- Do not attribute work to an AI in pull request descriptions, issues, or changelogs either.
- Author and co author trailers are for the people who did the work.

## Reviews and issues

- Follow the writing rules above in issues, reviews, and pull request descriptions.
- Prefer concrete, verifiable statements. Point at the code or the test rather than describing it in vague terms.
- If a change removes a capability, say why in plain terms, for example that no working in browser method exists for it.
