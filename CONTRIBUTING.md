# Contributing to Doot Doot Revolution

Thanks for helping. This is a small, deliberate codebase and the bar is that a
newcomer can read it top to bottom without a guide. Please keep it that way.

## Setup

Requires Node 20 or newer.

```
git clone https://github.com/virgilvox/doot-doot-revolution
cd doot-doot-revolution
npm install          # installs every workspace at once
npm run dev          # web dev server at http://localhost:4318
```

## Everyday commands

```
npm run dev            # web dev server
npm run dev:desktop    # the same app inside an Electron window
npm test               # unit tests across the logic packages (node --test)
npm run test:web       # Vue component and store tests (Vitest)
npm run test:all       # both suites
npm run build          # static web build to apps/web/dist
npm run build:desktop  # Electron installers for the current OS
```

## How the project is laid out

Sixteen framework-agnostic libraries under `packages/` hold all the game logic and
rendering, published under the `@doot-games` scope. The Vue app under `apps/web` is
a thin layer of views, composables, and stores over those packages, and also
packages as an Electron desktop app from the same renderer. `ARCHITECTURE.md`
explains the layers and the one rule that keeps them clean. Each package has its
own README describing its public interface.

The dependency direction is strict and enforced by review: logic packages depend
only on other logic packages, never on the DOM or on the app. Nothing under
`packages/` imports `apps/web`. If a change makes a package reach toward the app or
the DOM, it belongs in the app instead.

## Making a change

1. Branch from `main`.
2. Keep the change small and cohesive. Put logic in a package and wiring in the
   app. Do not add a framework where a function will do.
3. If you touch a package's public interface, update its README in the same change.
4. Add or update tests. Logic changes need a package test; view or store changes
   need a Vitest test.
5. Run `npm run test:all` and `npm run build` before you open the pull request.

## Pull request checklist

- [ ] `npm run test:all` passes.
- [ ] `npm run build` succeeds.
- [ ] Touched packages have updated READMEs.
- [ ] No emojis anywhere: code, comments, UI, docs, or commit messages.
- [ ] No em dashes and no double hyphens used as dashes in prose.
- [ ] No capability is shipped without a working implementation.
- [ ] Commit subjects are imperative ("add hold grace", not "added").
- [ ] No AI attribution of any kind: no "generated by" lines and no assistant
      co-author trailers in commits, PRs, or changelogs.

`RULES.md` is the full, binding list; the checklist above is the short version.

## Style

Plain JavaScript, no TypeScript. Vue single-file components use `<script setup>`.
The logic packages are intentionally terse; the app layer should read cleanly, with
comments that explain why rather than what. Match the surrounding code.

## Reporting bugs and ideas

Open an issue with steps to reproduce (for a bug) or the problem you want solved
(for a feature). Small, focused pull requests are welcome without a prior issue.
